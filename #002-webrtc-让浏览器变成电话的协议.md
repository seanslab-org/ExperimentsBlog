# WebRTC：让浏览器变成电话的协议

*Google 如何花了 6800 万美元，开源了一个瑞典电信引擎，引发了一场编解码器战争，并意外地为全球大流行构建了基础设施。*

---

2020 年 3 月，世界封锁了。一夜之间，每一所学校、每一家公司、每一家医院都需要视频通话。Zoom 的日会议参与者在三个月内从 1000 万飙升到 3 亿（这是 Zoom 自己的统计口径，同一用户参加多个会议会被重复计数——但即使考虑到这一点，增长曲线仍然惊人）。Google Meet 突破 1 亿。Microsoft Teams 爆炸式增长了 894%。

大多数人以为这是硅谷产品工程的胜利。并不是。这是一个协议的胜利——一个在 IETF 经过十年残酷争斗才悄然标准化的协议，一个源自以 6820 万美元收购一家大多数人从未听说过的瑞典小公司的协议，一个让你的浏览器做到浏览器从未被设计来做的事情的协议：发送实时音视频，点对点，亚秒级延迟，端到端加密。

这个协议叫 **WebRTC**。这是它诞生的故事——那些人、那些政治、那些塑造了它的工程决策，以及为什么它比大多数开发者意识到的更加重要。

---

## 第一章：瑞典的引擎室（1999–2010）

WebRTC 的真正故事不是从 Google 开始的。它始于斯德哥尔摩，1999 年，一家叫 **Global IP Sound** 的公司——后来更名为 **Global IP Solutions**，简称 **GIPS**。

GIPS 由一群信号处理研究者创立——**Roar Hagen**、**Bastiaan Kleijn**、**Espen Fjogstad** 和 **Ivar T. Hognestad**——他们对一个将定义接下来二十年电信业的问题有着清晰的认识：分组网络上的语音质量是一个信号处理问题，不是一个网络问题。

1999 年，VoIP 糟透了。不是因为网速慢——确实慢，但那不是真正的问题。真正的问题是分组网络对实时音频从根本上就是敌对的。数据包会乱序到达、延迟到达、或者根本不到达。发送端和接收端之间的时钟漂移导致缓冲区下溢。扬声器泄漏回麦克风的回声制造反馈环路。背景噪音遮蔽语音。而人耳对这一切极其敏感——即使 150 毫秒的单向延迟就会让对话感觉不自然。超过 400 毫秒，几乎无法使用。

GIPS 用定制的信号处理逐一攻克了这些问题。他们打造了两个在 VoIP 世界里成为传奇的编解码器：

**iLBC**（Internet Low Bitrate Codec）——一个为极端丢包恢复而设计的窄带语音编解码器。当大多数编解码器在 5–10% 的丢包率下就崩溃时，iLBC 在 20–30% 的丢包率下仍然可以听清。诀窍在于每一帧都可以独立解码——没有帧间依赖，所以丢失一个数据包不会损坏后续的音频。2004 年作为 RFC 3951 发布。

**iSAC**（Internet Speech Audio Codec）——一个宽带自适应编解码器，根据网络状况动态调整码率和帧大小。在大多数编解码器使用固定码率的时代，iSAC 持续测量丢包率和延迟并实时适应。如果你的网络恶化了，iSAC 会优雅降级，而不是碎裂成数字噪音。

但编解码器只是 GIPS 特殊之处的一部分。他们还构建了音频工程师所说的**"媒体引擎"**——完整的声学回声消除、噪声抑制、自动增益控制和抖动缓冲堆栈，位于原始麦克风输入和编码比特流之间。这就是将"喂，你能听到我吗？"的体验与"我们需要重新发起通话"的体验区分开来的关键。这是极其不光鲜的信号处理工作——那种你凌晨三点调试滤波器系数的活，因为回声消除器在某个特定型号的 Dell 笔记本上产生了振铃——而 GIPS 在这方面是世界上最好的。

他们的客户名单证明了这一点：Yahoo!、WebEx（后来的 Cisco）、AOL，以及——关键的是——Google Talk。Skype 有自己的引擎（通过 SILK），但 GIPS 驱动了 VoIP 行业的大部分。

**2010 年 5 月**，Google 出手了。他们以 **6820 万美元现金**报价收购 GIPS。交易于 2011 年 1 月完成。

6820 万美元，Google 拿到了编解码器、回声消除器、噪声抑制器、抖动缓冲器，以及打造这一切的整个瑞典信号处理工程师团队。就接下来发生的事情而言，这是 Google 有史以来花得最值的 6820 万美元。

## 第二章：开源（2011）

Google 买 GIPS 不是为了让 Google Talk 变得更好。他们买它是为了做一件更有野心的事：**把实时通信放进地球上的每一个浏览器**。

Chrome 团队发现了 Web 平台的一个关键缺口。到 2009 年，浏览器几乎能做原生应用能做的一切：复杂图形（Canvas、WebGL）、离线存储（IndexedDB）、后台处理（Web Workers）、双向通信（WebSockets）。但实时音视频？那还需要插件——Flash、Java 或专有浏览器扩展。而插件正在消亡。

愿景很简单：如果浏览器本身就能从麦克风和摄像头捕获音视频，使用高质量编解码器编码，协商一个穿透 NAT 和防火墙的点对点连接，加密媒体流，并以低于 200 毫秒的延迟交付——所有这些都通过一个 JavaScript API 实现呢？

这个愿景就是 WebRTC。

**2011 年 5 月 31 日**，**Harald Alvestrand**——Google 的挪威资深互联网标准架构师，前 IETF 主席——在 W3C public-webrtc 邮件列表上发布了一条消息，宣布 WebRTC 源代码的开源发布。代码发布在 webrtc.org。GIPS 的编解码器——iSAC、iLBC 及其回声消除和噪声抑制堆栈——与新的 VP8 视频编解码器集成以及完整的 ICE（交互式连接建立）NAT 穿越实现一起开源。

两个工作组同时成立：**W3C WebRTC 工作组**（负责浏览器 JavaScript API）和 **IETF RTCWeb 工作组**（负责底层协议）。双轨方式是刻意的——Google 知道，没有协议标准化的浏览器 API 是无用的，而没有浏览器 API 的协议标准化是学术的。

**Justin Uberti**，一位曾主导 AIM、Gmail 视频聊天和 Google Hangouts 视频技术的 Google 工程师，成为了工程负责人。Uberti 是那种罕见的既懂信号处理层又懂浏览器 API 设计层的工程师——他可以争论声学回声消除滤波器拓扑结构，然后转身设计一个 Web 开发者真正能用的 JavaScript API。

这个雄心令人屏息。但天真也同样如此。Google 以为标准化需要两三年。实际上花了十年。

## 第三章：编解码器战争（2012–2014）

第一场大战是关于**强制实现（MTI）**视频编解码器——每个符合 WebRTC 标准的浏览器都必须支持的编解码器。这听起来像是一个枯燥的技术决策。事实上，它是地球上一些最大公司之间的代理战争。

**Google** 要 **VP8**，这是它 2009 年宣布收购 On2 Technologies（2010 年完成交易，花了 1.246 亿美元——Google 当时正在疯狂收购）后获得的视频编解码器，并作为 WebM 项目的一部分开源。VP8 是免版税的，这符合 Google 的开放网络哲学，而且——也不是巧合——意味着 Google 不需要向竞争对手支付许可费。

**Apple**、**Microsoft**、**Cisco** 以及大部分传统电信行业想要 **H.264**，世界上占主导地位的视频编解码器。H.264 无处不在——在每部智能手机、每个蓝光播放器、每个视频会议系统、每个硬件编码器芯片里。它有大量的硬件加速支持。但 H.264 通过 **MPEG-LA** 专利池许可，使用它需要支付版税。

IETF RTCWeb 邮件列表爆发了只能称之为多年火焰战的争吵。论点混合了合理的技术关切和赤裸裸的企业战略：

**VP8 阵营认为：**VP8 是免版税的。一个面向 Web 的开放标准不应该要求支付专利版税。H.264 的许可费用，虽然对大公司来说可以承受，但对较小的开发者和开源项目来说是一个障碍。一个受版税约束的强制编解码器与开放网络的理念背道而驰。

**H.264 阵营认为：**VP8 在技术上不如 H.264（有争议，但在当时是站得住脚的）。H.264 有普遍的硬件支持——每部手机、每个机顶盒都在硅片中内置了 H.264 解码。VP8 的硬件支持在 Google 自己设备之外基本不存在。强制要求 VP8 会迫使整个硬件生态系统为一个只有一家公司在推动的编解码器添加支持。而且关于 VP8 自身的专利风险还有悬而未决的问题——MPEG-LA 曾公开探索创建 VP8 专利池。

辩论持续了两年多。一点都不好看。正如一位参与者指出的，房间里大多数人从一开始就太有偏见了，而那些没有偏见的人觉得自己没有足够的信息来做决定。

**2014 年 11 月**，IETF 工作组达成了一个所罗门式的妥协：**VP8 和 H.264 都是强制实现的**。每个 WebRTC 实现都必须支持两种编解码器。这被广泛视为 Google 得到了它想要的（VP8 被标准化），同时电信行业也得到了它想要的（H.264 被强制）。这到底是优雅的妥协还是怯懦的回避，取决于你问谁。

实际结果是：浏览器最终都支持了两种，世界继续前行。Google 后来推动了 VP9，然后是 AV1（通过开放媒体联盟开发，由 Google、Mozilla、Cisco、Microsoft、Netflix、Amazon 和 Intel 于 2015 年联合创立），编解码器战争仍在继续——但 WebRTC MTI 之战是其中政治色彩最浓的一场。

## 第四章：SDP 问题与 ORTC 叛乱

如果说编解码器战争是最公开的争斗，那么 **SDP 辩论**就是技术上最具后果性的争斗。

WebRTC 从电话世界继承了 **SDP**（会话描述协议）。SDP 是一种基于文本的格式，最初于 1998 年定义，用于描述多媒体会话——有哪些编解码器可用、使用哪些 IP 地址和端口、协商哪些加密参数。在 WebRTC 中，两个对等方交换 SDP"offer"和"answer"来建立通话。这就是 **offer/answer 模型**，直接来自 SIP（会话发起协议），VoIP 电话的标准协议。

问题在于 SDP 是为静态的、预定义的媒体会话世界设计的。WebRTC 需要的东西要动态得多。想在现有通话中添加屏幕共享？重新协商整个 SDP。想改变视频分辨率？重新协商。想从发送切换到接收？重新协商。每次更改都需要完整的 offer/answer 交换，这意味着往返延迟和潜在的眩光（双方同时尝试重新协商）。

SDP 本身也是——委婉地说——一种只有电信工程师才能爱上的格式。一个典型的 WebRTC SDP 块是数百行分号分隔的属性，分布在多个"m-line"中，每个都有自己的参数集、扩展和 ICE 候选。解析和生成 SDP 是脆弱的、容易出错的，也是 WebRTC 互操作性 bug 中不成比例的来源。

**2013 年**，以 **Iñaki Baz Castillo** 为首、微软支持的一群开发者提出了 **ORTC**（Object Real-Time Communication）——一个根本不同的 API，将 WebRTC 堆栈分解为离散的、可独立控制的对象：`RTCIceGatherer`、`RTCIceTransport`、`RTCDtlsTransport`、`RTCRtpSender`、`RTCRtpReceiver`。没有 SDP。没有 offer/answer。只有你用编程方式连接在一起的对象。

ORTC 的设计更干净、更灵活，更符合 Web 开发者对 API 的思维方式。它的杀手级优势：**不需要重新协商**。想添加一个视频轨道？创建一个新的 `RTCRtpSender` 并将其附加到现有传输层。ICE 和 DTLS 层不关心——它们已经建立了。只需在带外传输新的 RTP 参数，就完成了。

微软实际上在 **Edge**（Chromium 之前的版本）中发布了 ORTC。它能用。但它是一个采用率有限的平行标准，当微软在 2018 年将 Edge 转向 Chromium 时，ORTC 支持实际上被抛弃了。

**WebRTC 1.0** 中出现的妥协是一个混合体：SDP offer/answer 模型仍然是主要的 API 表面，但添加了较低层次的对象如 `RTCRtpSender` 和 `RTCRtpReceiver`，给开发者提供了 ORTC 所承诺的一些编程控制。不够优雅，但它发布了。实用主义者赢了。

## 第五章：架构解析（给真正好奇的人）

WebRTC 不是一个单一协议。它是一个**协议栈**，每一层解决一个不同的问题，以特定方式串联在一起。理解这个栈就是理解为什么 WebRTC 能工作得这么好——以及为什么它这么复杂。

### 第一层：连接性——ICE、STUN、TURN

WebRTC 在网络层解决的根本问题是 **NAT 穿越**。互联网上的大多数设备都在 NAT（网络地址转换器）后面，这意味着它们有私有 IP 地址，无法从公共互联网直接路由。两个在不同 NAT 后面的对等方不能简单地相互打开一个 socket。

**ICE**（交互式连接建立，RFC 8445）是协调这个过程的框架。它从三个来源收集"候选"——潜在的地址/端口组合：

- **主机候选**：设备的本地 IP 地址。
- **服务器反射候选**：STUN 服务器看到的设备的公网 IP 地址。
- **中继候选**：TURN 服务器上将中继流量的地址。

然后 ICE 执行**连接性检查**——它尝试每一对可能的本地和远程候选组合，发送 STUN 绑定请求来测试可达性。候选对按优先级排序（主机到主机优先于中继），选择最佳的可用候选对。

**STUN**（NAT 的会话穿越工具，RFC 8489——取代了最初的 RFC 5389）是一个用于发现你的公网地址的轻量级协议。STUN 服务器就像一面镜子——你给它发一个包，它告诉你这个包是从什么源 IP 和端口到达的。这就是 NAT 后面的设备如何得知自己的服务器反射候选的。

**TURN**（使用中继穿越 NAT，RFC 8656——取代了 RFC 5766）是后备方案。当直接连接不可能时——通常是因为一个或两个对等方在对称 NAT 后面——流量通过 TURN 服务器中继。这增加了延迟并且需要花钱（得有人运行那些服务器），但它保证了连接性。在实践中，大约 15–20% 的 WebRTC 会话需要 TURN 中继。

对于任何在嵌入式系统上调试过连接问题或在移动网络上处理过运营商级 NAT 的人来说，ICE 的优雅值得欣赏。它系统地尝试每一条可能的路径，从最便宜和最直接的开始，然后优雅地回退。这种协议看起来过度工程化了，直到你意识到每一个分支都存在，因为某个特定的 NAT 拓扑打破了更简单的方案。

### 第二层：安全——DTLS 和 SRTP

所有 WebRTC 媒体都是加密的。这不是可选的——这在规范中是强制的。加密架构使用两个协议：

**DTLS**（数据报传输层安全，RFC 6347）——本质上是 UDP 版的 TLS。它在对等方之间执行握手以建立共享加密密钥，并使用自签名证书进行身份验证。证书指纹通过信令通道（通常通过 SDP）交换，这样每一方都能验证对方的身份。

**SRTP**（安全实时传输协议，RFC 3711）——实际加密媒体数据包的协议。SRTP 使用 DTLS 协商的密钥（这种特定组合称为 **DTLS-SRTP**，定义在 RFC 5764 中）。

IETF 强制要求 DTLS-SRTP 并明确禁止了 **SDES**（会话描述协议安全描述），一种将会话密钥直接嵌入 SDP offer/answer 的旧密钥交换机制。SDES 更简单、更快——不需要握手——但它将媒体加密密钥暴露在信令路径上，这在浏览器环境中意味着任何能访问 SDP 的 JavaScript 都能提取密钥。更糟糕的是，任何触及信令的中间方（你的信令服务器、你的网关、你的企业代理）都能静默解密所有媒体。这是一种让密码学家皱眉的安全模型。禁止 SDES 的决定在已经围绕它构建了基础设施的电信厂商中引起了争议，但这是正确的选择。

### 第三层：媒体——RTP、RTCP 和媒体引擎

实际的音视频数据通过 **RTP**（实时传输协议，RFC 3550）流动，**RTCP**（RTP 控制协议）伴随其运行以提供反馈——丢包统计、往返时间估算、接收方报告。WebRTC 使用 RTCP 进行拥塞控制，让发送方根据网络条件调整码率。

协议层之下是**媒体引擎**——实际捕获、处理、编码和解码音视频的软件。这就是 GIPS 收购得到回报的地方。WebRTC 媒体引擎包括：

- **声学回声消除（AEC）**：消除扬声器播放回到麦克风的回声。这是一个非平凡的自适应滤波器问题——当有人移动笔记本电脑、在椅子上挪动、或在桌上放一杯咖啡时，回声路径都会改变。
- **噪声抑制（NS）**：估计噪声底噪并从信号中减去它。挑战在于区分噪声和低电平语音——太激进会截断字词；太温和噪声就会泄漏。
- **自动增益控制（AGC）**：归一化音频电平，使耳语的人和大声说话的人以相似的音量传出。
- **抖动缓冲**：缓冲传入的数据包以平滑到达时间的变化。太小会有断音；太大会增加延迟。最优大小取决于持续变化的网络条件。

对于那些在 DSP 硬件上花过时间调整自适应滤波器或调试 AGC 压缩器曲线的人来说，WebRTC 的音频流水线工程化得非常出色。GIPS 团队在软件中解决了这些问题，质量堪比专用 DSP 实现。仅回声消除器就包含了多年积累的专业知识——处理双讲、非线性回声路径，以及笔记本扬声器距内置麦克风仅几英寸时的特殊噩梦。

### 第四层：数据——SCTP over DTLS

WebRTC 还支持**数据通道**——对等方之间的任意双向数据流。它们运行在通过 DTLS 隧道传输的 **SCTP**（流控制传输协议）之上。数据通道可以配置为可靠的（重传丢失的数据包）或不可靠的（发射后不管，类似 UDP），有序或无序，你可以在单个对等连接上打开多个独立通道。

这就是让 WebRTC 不仅仅是一个电话的东西。数据通道驱动文件传输、聊天、游戏状态同步以及任何其他实时数据交换。它们使用媒体栈已经建立的相同 ICE/DTLS 连接，所以没有额外的 NAT 穿越开销。

## 第六章：十年标准化马拉松

WebRTC 的标准化时间线是一个关于互联网标准实际上如何制定的案例研究——也就是说，缓慢地、痛苦地，更多靠精疲力竭而非共识。

**2011 年**：Google 开源代码。W3C 和 IETF 工作组成立。乐观情绪高涨。"我们两年就能搞定。"

**2012 年**：Chrome 在实验性标志后面发布初始 WebRTC 支持。Firefox 跟进。编解码器战争开始。

**2013 年**：ORTC 作为替代 API 被提出。SDP 辩论加剧。Chrome 和 Firefox 之间的互操作性充其量是脆弱的。

**2014 年**：编解码器 MTI 妥协（VP8 和 H.264 都要支持）。Chrome 和 Firefox 大致上可以互通了，但边缘情况比比皆是。Safari 引人注目地缺席。

**2015–2016 年**：规范缓慢而艰苦地推进。提交了数百个 issue。SDP 语义逐行争论。

**2017 年**：**Apple 在 Safari 11 中发布 WebRTC 支持。** 这是分水岭时刻。有了 Safari 加入，WebRTC 在每个主要浏览器中都可用了。三浏览器问题（Chrome、Firefox、Safari）解决了。Apple 的实现最初是有限的和有 bug 的，但它存在——这就够了。

**2018 年**：微软将 Edge 转向 Chromium，有效地杀死了 ORTC，并将浏览器格局整合到一个单一的 WebRTC 实现（Google 的）周围。

**2021 年**：**WebRTC 1.0 成为正式的 W3C 推荐标准**——距离最初发布整整十年。IETF 发布 RFC 8825–8835，正式定义了协议栈。以互联网标准的时间尺度来说，十年并不罕见。但对于一个已经被数十亿用户在生产中使用的技术来说，现实与规范之间的差距是令人瞩目的。

## 第七章：大流行压力测试

WebRTC 最初是为浏览器之间的一对一通话设计的。COVID-19 把它变成了人类文明的关键基础设施。

这个协议的设计——点对点、低延迟、浏览器原生——使它成为视频会议爆发的天然基础。但从两人通话扩展到 2020 年的需求，需要远超原始规范的创造性工程。

**Google Meet** 在客户端使用 WebRTC，但将所有媒体路由通过 Google 自己的选择性转发单元（SFU）——接收每个参与者的媒体并有选择地转发给其他人的服务器。这避免了 N×N 网格拓扑问题（每个参与者将自己的流发送给每个其他参与者），并给了 Google 对带宽分配、质量自适应和录制的控制权。

**Discord** 从一开始就在其语音聊天中使用 WebRTC，最初是为需要超低延迟语音的游戏玩家设计的。当大流行来袭时，Discord 突然变成了一个拥有 1.4 亿月活用户的社交平台，承载从虚拟教室到观影派对的一切。

**Zoom** 是一个有趣的例外。Zoom 的核心架构**不是** WebRTC——它使用专有的媒体流水线和自己的编解码器和网络传输。但 Zoom 的网页客户端（在浏览器中加入而不下载应用）确实使用 WebRTC，Zoom 的工程师也承认借鉴了 WebRTC 的拥塞控制和连接建立模式。

**Jitsi**，由 **Emil Ivov** 构建的开源视频会议平台，成为了那些不能或不愿使用商业平台的组织的首选自托管替代方案。Jitsi 是纯 WebRTC、开源的，在大流行期间，它每月处理数亿分钟的视频会议。

大流行揭示了 WebRTC 的优势和局限。该协议非常有弹性——它处理了流量的爆发而没有根本性崩溃。但从"浏览器到浏览器点对点"到"50 人视频会议加屏幕共享、录制、分组讨论室和实时转录"之间的差距需要 WebRTC 本身不提供的基础设施层。SFU 模式、联播（发送多种质量级别）和可伸缩视频编码——这些都是建立在 WebRTC 之上的工程模式，而不是核心规范的一部分。

## 第八章：那些人

WebRTC 由一个异常多元的群体塑造——电信老兵、浏览器工程师、IETF 元老和创业公司创始人：

**Harald Alvestrand**——共同主持 W3C WebRTC 工作组的挪威标准老兵。Alvestrand 曾担任 **IETF 本身的主席**（2001–2005），这给了他在互联网两个最重要的标准机构中同时牧引一个标准的制度信誉。他的耐心是传奇的——也是必需的。

**Justin Uberti**——弥合信号处理和 Web API 之间鸿沟的工程负责人。Uberti 做了关于浏览器 RTC API 该是什么样的早期探索，利用了他构建 AIM 视频聊天和 Google Hangouts 的经验。他是在标准化过程威胁要分裂技术愿景时将其凝聚在一起的人。

**Serge Lachapelle**——Uberti 在 Google 的产品管理搭档，也是瑞典视频会议公司 **Marratech** 的联合创始人，Marratech 于 2007 年被 Google 收购。Lachapelle 是通过那次收购进入 Google 的，这意味着他从产品侧理解实时通信——不仅知道如何构建一个媒体引擎，更知道为什么 Google 应该花 6800 万美元把它送出去。

**Philipp Hancke**——一位德国工程师，成为 WebRTC 社区最多产的贡献者和调试者之一。Hancke 的博客文章和对 WebRTC 内部机制的分析——尤其是他对 SDP 和互操作性问题的解剖——成为任何在生产中实现 WebRTC 的人的必读材料。

**Emil Ivov**——Jitsi 的创始人，Google 自身实现之外最重要的开源 WebRTC 项目。Ivov 证明了 WebRTC 可以作为一个功能齐全的视频会议系统的基础，不需要任何专有组件。

**Iñaki Baz Castillo**——共同撰写 ORTC 规范并热情地论证 SDP 是 WebRTC 错误基础的西班牙开发者。历史部分地证明了他是对的——他所倡导的低层对象（RTCRtpSender、RTCRtpReceiver）进入了 WebRTC 1.0，即使完整的 ORTC 愿景没有实现。

在他们所有人的背后，是**来自斯德哥尔摩的 GIPS 工程师**——信号处理专家，他们的声学回声消除器、噪声抑制器和自适应抖动缓冲器位于每一个 WebRTC 通话的最底层，让实时音频的原始物理学真正地工作。他们的名字大多不在标准文件和博客文章中出现。他们的代码在地球上的每一个浏览器里。

## 第九章：WebRTC 到底改变了什么

在 WebRTC 之前，构建一个实时通信应用需要：

1. 一个原生应用（或一个 Flash/Java 插件）。
2. 许可一个专有媒体引擎。
3. 构建或购买一个 NAT 穿越方案。
4. 实现加密（或者，更常见的是，不实现加密）。
5. 通过应用商店或安装程序分发你的应用。

在 WebRTC 之后，你需要：

1. 一个网页。
2. 大约 50 行 JavaScript。
3. 一个 STUN 服务器（Google 提供免费的）。

从"数百万美元的工程项目"到"周末黑客马拉松"的这个差距——这才是 WebRTC 影响力的真正衡量标准。它将实时通信民主化了，就像 HTML 民主化了出版、WebSockets 民主化了双向数据一样。

今天，WebRTC 运行在超过 **40 亿个浏览器** 中，跨越 Chrome、Firefox、Safari 和 Edge。它驱动着 Google Meet、Facebook Messenger、WhatsApp Web、Discord、Slack huddles、Amazon Chime、Twitch 互动直播、远程医疗平台、远程控制机器人、实时游戏串流、基于浏览器的电话系统，以及数以千计的应用——这些应用在没有 WebRTC 的情况下要么不可能，要么成本高得令人望而却步。

而且不像大多数诞生于 Google 的协议，WebRTC 是一个真正的开放标准，由 Mozilla、Apple 和 Microsoft（通过 Chromium）独立实现。Google 在 2011 年开源的 GIPS 代码已经被来自各组织的数千名贡献者重写、扩展和优化。它现在属于 Web 了。

## 尾声：仍在响铃，仍在进化

WebRTC 在 2021 年 1 月成为正式的 W3C 推荐标准——距离 public-webrtc 邮件列表上的第一次宣布整整十年。它经历了一场编解码器战争、一次关于 SDP 的哲学分裂、ORTC 的慢动作消亡、Apple 最初拒绝参与，以及一场全球大流行的突然而残酷的压力测试。

下一个篇章已经在书写。**WebTransport** 正在作为 WebRTC 的补充出现，用于服务器到客户端的实时分发。**Insertable Streams** 现在允许即使通过 SFU 服务器也能实现端到端加密。**WebCodecs** 给开发者直接访问硬件编码器和解码器的能力，完全绕过 WebRTC 的托管媒体流水线。**AV1** 正在成为占主导地位的视频编解码器，WebRTC 实现正在添加硬件加速的 AV1 支持。

但核心成就不变：一群瑞典信号处理工程师构建了一个媒体引擎，Google 买下它并把它送出去了，十年激烈的标准化将它变成了十亿人用来度过大流行的基础设施。

下次你加入 Google Meet 通话时，音频就是好使——没有回声、没有噪音、没有"喂你能听到吗"——请记住，有一个抖动缓冲器在实时适应，有一个回声消除器在追踪你房间的脉冲响应，有一个噪声抑制器在估计你环境的频谱底噪。那不是魔法。那是二十五年瑞典 DSP 工程，开源给了全世界。
