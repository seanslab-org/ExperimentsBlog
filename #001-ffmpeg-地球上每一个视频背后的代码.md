# FFmpeg：地球上每一个视频背后的代码

*以及打造它的那群疯狂的人。*

---

如果你看过 YouTube 视频、在 Netflix 上追过剧、用 HandBrake 转过格式、或者用 VLC 播放过任何东西——恭喜，你已经用过 FFmpeg 了。你只是不知道而已。

FFmpeg 是整个多媒体世界的隐形管道。它在你的手机里、浏览器里、智能电视里、游戏机里，甚至——这不是开玩笑——在 NASA 的毅力号火星车上。将近 3,000 家公司在生产环境中使用它。然而，维护它超过十年的那个人，选择了贫困的生活，只为能全职写这些代码。

这是 FFmpeg 的故事。一个关于天才、自我、化名、分裂战争、耻辱堂和数十亿美元行业价值的故事——而这一切都建立在几乎没有人付费的代码之上。

---

## 第一章：一个用假名字的法国人（2000）

2000 年 12 月 20 日。一个程序员向公开仓库推送了一个新多媒体框架的第一个 commit。作者名字写的是：**Gérard Lantau**。

这名字是假的。

真正的作者是 **Fabrice Bellard**，一位 1972 年出生于法国格勒诺布尔的计算机科学家。Bellard 使用化名是因为在 2000 年，自己写视频编解码器实现是一种被专利持有者起诉的绝佳方式。多媒体专利的版图就是一片雷区——MPEG-LA 正在积极地许可和执法，而一个开源编解码器库基本上就是在自己背上画靶子。

所以 Bellard 做了任何一个理性的天才都会做的事：照样发布了代码，只是用了假名字。

项目叫 **FFmpeg**——"FF"代表 **Fast Forward**（快进），"mpeg"来自动态图像专家组标准。动机简单而务实：在 1999 年的 Linux 上，如果你下载了一个视频然后双击它，*什么都不会发生*。你需要 RealPlayer、QuickTime 或 Windows Media Player，而这些都不原生运行在 Linux 上。多媒体栈是碎片化的、专有的，对开放平台充满敌意。

Bellard 的解决方案一如既往地优雅。他将 FFmpeg 设计为**编解码器层**（`libavcodec`）和**格式/容器层**（`libavformat`）的清晰分离。这个架构——事后看来似乎理所当然——在当时是革命性的。它意味着你可以自由组合：从 AVI 容器解码 H.263、编码 MPEG-2 到传输流、或者在任意编解码器和格式之间转码。到 2001 年，FFmpeg 已经能处理 MPEG-1、MPEG-2、MPEG-4、H.263 和多种音频编解码器。

### Fabrice Bellard 到底是谁？

如果你是音视频开发者，你肯定知道 FFmpeg。但你可能不知道，同一个人还创造了：

- **QEMU**——处理器模拟器，几乎支撑着所有现代云虚拟化。AWS、Google Cloud 和 Azure 使用的 KVM 就是建立在 QEMU 之上的。
- **Tiny C Compiler (TCC)**——一个小到能极速编译 Linux 内核的 C 编译器。
- **Bellard 公式**——一个计算圆周率单个数位的数学公式。2009 年，他用一台普通台式机算出了 **2.7 万亿位圆周率**，创下当时的世界纪录。
- 他还多次赢得国际混乱 C 代码大赛（IOCCC），因为显然只在有用的方面做天才还不够。

Bellard 有一个模式：创造改变世界的东西，把它做到能自我维持的程度，然后转身去造下一个不可能的东西。FFmpeg 也不例外。

## 第二章：Michael Niedermayer 登场（2004）

到 2003-2004 年，Bellard 已经对 FFmpeg 的日常维护失去兴趣。更大的事情在召唤他（QEMU 在等着）。于是在 2004 年，他把项目交给了 **Michael Niedermayer**，一位来自 MPlayer 社区的奥地利开发者。

Niedermayer 不仅仅是在维护 FFmpeg——他*成为了* FFmpeg。他编写或大幅改进了关键组件：视频后处理库、`libswscale`（图像格式转换库）以及无数编解码器实现。他高产、技术精湛，对项目完全奉献。

有多奉献？**Niedermayer 选择以最低收入生活，这样他就不用把时间浪费在其他工作上。** 他很少参加技术会议，没有追求传统的职业道路，把一切都献给了 FFmpeg。

对于音视频开发者来说，这就是你默默依赖了二十年的那种人。每次你运行 `ffmpeg -i input.mp4 output.avi`，你都在运行他多年心血的结晶。

但 Niedermayer 的领导风格……两极分化。他的态度是"我想 commit 什么就 commit 什么"，不太在意代码规范，偏好邮件列表而非面对面讨论，对新人可能比较粗暴。随着项目的成长，这些特质从小摩擦变成了严重的治理问题。

---

## 第三章：分裂大战（2011）

**2011 年 3 月 13 日**，一群核心 FFmpeg 开发者做了一件戏剧性的事：他们 fork 了整个项目，创建了 **Libav**。

原因与其说是技术分歧，不如说是人的问题。Libav 阵营的不满集中在 Niedermayer 的集中式领导模式上——他处理 commit 权限、代码审查和项目管理的方式让许多贡献者觉得武断和压抑。紧张局势大约从 2010 年开始积累，但这次 fork 让一切爆发了。

接下来发生的事情非常混乱，即使以开源圈的戏剧标准来衡量也是如此：

- Libav 开发者一度**夺取了 ffmpeg.org 域名的控制权**（原始所有者 Bellard 最终帮助收回了它）。
- Libav 说服了多个主要 Linux 发行版——**Debian、Ubuntu 和 Gentoo**——将默认包从 FFmpeg 切换为 Libav。
- 两个项目继续并行开发，补丁有时在两边流动（有时不流动，导致相同功能出现不同实现）。

很多年里，如果你在 Debian 系统上运行 `ffmpeg`，你实际上运行的是 Libav 的二进制文件，而不是 FFmpeg 的。大多数用户完全不知道。

**2015 年 6 月**，Niedermayer 自己辞去了 FFmpeg 项目负责人的职位，项目转向了更加分布式的治理模式。但那时候，局势已经扭转：FFmpeg 的社区重新集结，各发行版开始切换回来，Libav 逐渐失去了势头。

**Libav 在 2020 年被宣布废弃。** FFmpeg 赢了。

但这场分裂战争留下了伤疤。它分裂了贡献者群体，造成了数年的重复劳动，并且一度真正威胁到了世界上最重要的多媒体软件的存续。

## 第四章：耻辱堂

FFmpeg 采用 **LGPL**（或 GPL，取决于编译选项）许可。这意味着如果你使用它，你需要遵守许可证——公开你的源代码，或至少在被要求时提供，并给予适当的署名。

猜猜有多少公司正确地做到了这一点？

FFmpeg 的回应令人愉快地对抗性十足：他们创建了**[耻辱堂](https://ffmpeg.org/shame.html)**——一个公开列出违反许可证的公司名单。那些悄悄把 FFmpeg 塞进商业产品里，既不给署名也不公开源码的视频转换工具、媒体播放器和流媒体应用，被直接点名示众。

一些违规者乖乖合规后被移出了名单。一些则置之不理。有一个违规者——一个商业媒体播放器的创始人——甚至**威胁** FFmpeg 开发者，就因为他们指出了违规行为。

对于任何做过开源软件的开发者来说，这都很有共鸣。你写了代码。一家公司给它套了个 GUI，卖 29.99 美元，假装是他们自己写的。耻辱堂就是 FFmpeg 社区表达态度的方式：*我们看到你了*。

## 第五章：世界运行在 FFmpeg 之上

我们来谈谈 FFmpeg 影响力的规模，因为怎么说都不过分。

**YouTube**——FFmpeg 的库是核心视频处理流水线的一部分。你上传的每一个视频，都经过 Bellard 在 2000 年开始的代码的后代处理。

**Netflix**——使用 FFmpeg 编码其整个视频库，为不同设备和网络速度进行优化。他们也是 SVT-AV1 的主要贡献者，这个下一代编码器已经集成到 FFmpeg 中。

**VLC**——世界上最流行的开源媒体播放器，底层由 `libavcodec` 和 `libavformat` 驱动。

**Chrome/Chromium**——Google 的浏览器使用 FFmpeg 进行媒体播放。如果你看过 HTML5 视频，FFmpeg 解码了它。

**Blender、HandBrake、Plex、Kodi、Shotcut、Bilibili**——全部建立在 FFmpeg 之上。

**NASA 毅力号火星车**——使用 FFmpeg 进行图像和视频压缩，然后再将画面从火星传回地球。

而讽刺的是：**Google 在 Chrome 和 YouTube 中使用 FFmpeg，2025 年收入超过 4,000 亿美元，最近却给 FFmpeg 发了一份 AI 生成的安全漏洞清单——既不提供补丁，也不提供资金支持。** 与此同时，维护这个代码库超过十年的人选择了贫困。

这就是开源可持续性问题最纯粹的形式。

## 第六章：架构解析（给技术控的）

如果你是音视频开发者，你已经领教过 FFmpeg 命令行的威力。但底下的架构值得深入了解。

FFmpeg 的核心库：

- **`libavcodec`**——编解码器库。包含数百种音视频格式的编码器和解码器。这是跳动的心脏。从古老的 RealVideo 到前沿的 AV1，这里应有尽有。
- **`libavformat`**——容器/解复用/复用库。处理 MP4、MKV、FLV、MPEG-TS、WAV、FLAC 等数十种格式的读写。
- **`libavfilter`**——滤镜图引擎。缩放、裁剪、叠加、色彩校正、音频重采样——如果你能把它描述为处理图，它就在这里。
- **`libswscale`**——像素格式转换和图像缩放。让 YUV420p 到 RGB 的转换飞快的那个东西。
- **`libswresample`**——音频重采样、格式转换和声道布局重混。
- **`libavutil`**——共享工具函数。数学运算、内存分配、像素格式描述符。

这个架构的精妙之处在于，YouTube、Netflix、Vimeo 这样的公司可以**直接链接 `libavcodec` 和 `libavformat`**，完全绕过 `ffmpeg` 命令行程序。这给了他们对线程、内存分配和 I/O 的精细控制——这正是大规模部署所需要的。

对于那些在 DSP 核心上手写 SIMD 优化长大的人来说，FFmpeg 的性能策略有一种深层的满足感：针对 x86（SSE、AVX）、ARM（NEON）和其他架构手写汇编，加上 C 语言回退保证可移植性。这就是那种有人对每个热循环都做了性能分析、让每个时钟周期都物尽其用的代码。

## 第七章：塑造 FFmpeg 的关键人物

除了 Bellard 和 Niedermayer，还有许多杰出开发者塑造了这个项目：

- **Måns Rullgård**——将 FFmpeg 的构建系统打磨到近乎完美。任何人在 FFmpeg 上运行 `./configure && make` 并且*一切正常工作*跨平台的时候，都应该感谢 Rullgård。
- **Baptiste Coudurier**——广播领域的专家。为 ProRes、DNxHD、MXF 以及其他对影视行业至关重要的格式带来了专业级支持。如果 FFmpeg 能在专业后期制作流水线中工作，Coudurier 是最大的功臣之一。
- **Loren Merritt**——FFmpeg 中 H.264 编解码器的维护者，也是 x264 开发的关键人物。他在最快开源 H.264 编码器上的工作为软件编码性能树立了标杆。
- 还有数百人——GitHub 上的 MAINTAINERS 文件很长，每个名字都代表一个足够热爱多媒体、愿意为公共事业做贡献的人。

## 尾声：依然在运行，依然缺钱

FFmpeg 在 2025 年迎来了 25 岁生日。它经历了化名创立、敌意 fork、治理危机、猖獗的许可证违规和长期资金不足。它仍然是世界上最重要的多媒体基础设施。

如果你是音视频开发者，你的工具链中的每一个工具，要么*就是* FFmpeg，要么*依赖于* FFmpeg。这是那种让你重新思考工程"影响力"含义的软件。不是创业公司，没有融资轮次，没有退出。只是一个库，由一个用假名字的人创建，最终运行了地球上的每一个视频。

下次你运行 FFmpeg 命令的时候，也许可以去 [ffmpeg.org/donations](https://www.ffmpeg.org/donations.html) 捐几块钱。代码是免费的。写代码的人还是要吃饭的。
